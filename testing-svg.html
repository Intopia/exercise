Testing SVG images inside links and buttons

Note: examples 1 and 6 in each section (no accessible name) demonstrate a computation baseline only. A link or button with no accessible name is a real accessibility failure and should never appear in production code.

Links

Link example 1: no name from link or SVG

<a href="#">
  <svg></svg>
</a>

Pass = No accessible name

Link example 2: aria-label from SVG

<a href="#">
  <svg aria-label="Hello"></svg>
</a>

Pass = Accessible name must be "Hello" from link content

Link example 3: title from SVG

<a href="#">
  <svg>
    <title>Hello</title>
  </svg>
</a>

Pass = Accessible name must be "Hello" from link content

Link example 4: aria-label on link

<a href="#" aria-label="Goodbye">
  <svg aria-label="Hello"></svg>
</a>

Pass = Accessible name must be "Goodbye" from aria-label

Link example 5: aria-label on link

<a href="#" aria-label="Goodbye">
  <svg>
    <title>Hello</title>
  </svg>
</a>

Pass = Accessible name must be "Goodbye" from aria-label

Link example 6: role="img" plus aria-label on SVG

<a href="#">
  <svg role="img" aria-label="Hello"></svg>
</a>

Note for testing: does adding role="img" change real-world AT exposure compared to Link example 2, which has no explicit role? Spec computation should produce the same accessible name either way, this tests whether current browser/AT combinations actually agree.

Link example 7: decorative SVG (aria-hidden) with visible text

Visit our site 
<a href="#">
  <svg aria-hidden="true"></svg>
  Visit our site
</a>

Note for testing: does the link's accessible name come from "Visit our site" only, with the SVG fully excluded from the computation?

Link example 8: aria-labelledby on SVG, referencing hidden text

Hello

<a href="#">
  <svg aria-labelledby="link-svg-label"></svg>
</a>

<p id="link-svg-label" class="hidden">Hello</p>

Note for testing: does aria-labelledby on an SVG behave the same way it does on other elements, including reaching through visually hidden text?
